## Overview

v1.6.0 of `Scan-PythonPackages.ps1`, a PowerShell-based static security scanner for Python packages submitted through the media transfer review workflow. The tool is designed for operator-driven inspection of untrusted Python artifacts before they are admitted to a trusted environment.

All analysis performed by this tool is **strictly static**. Submitted package code is not executed, notebooks are not executed, and notebook kernels are never started. Jupyter notebook code cells are copied into temporary Python source projections so the existing static analyzers can inspect notebook code without changing the scanner's trust boundary.

> **Note on version history.** The script went through several internal drafts (`1.0`, `1.0-PEP440`) before the v1.2 GA release, which established the stable installation layout (venv, logs, staging, and reports all in separate locations). v1.3 added the Pester test harness (`tests/`). v1.4 added unsupported-file detection. v1.5 replaces the presence-only native binary check with triage-level PE/ELF binary inspection and migrates the project layout to git with canonical paths. v1.5.1 cleaned up CI permissions and scanner bootstrap upgrades. v1.5.2 corrects release packaging and documentation so the required Python helper is distributed with the scanner. v1.6.0 adds first-class Jupyter notebook (`.ipynb`) discovery and static code-cell scanning.

### What's New in v1.6.0

v1.6.0 adds first-class Jupyter notebook (`.ipynb`) support. This closes GitHub issue #1, where notebooks were previously surfaced only as unsupported files. That behavior provided visibility that a notebook was present, but it did not inspect notebook code cells where malicious Python could hide.

The scanner now recognizes notebooks in two places:

- **Loose notebooks** directly under the submitted scan folder.
- **Embedded notebooks** inside supported archives after extraction to the temporary staging workspace.

Notebook handling remains static and conservative:

- Notebook JSON is parsed with PowerShell's structured JSON parser.
- Only cells where `cell_type` is `code` are copied into temporary `.py` projection files.
- Markdown cells are not copied into the Python projection.
- Saved outputs and attachments are surfaced as `NotebookParser` findings for manual review because they are visible notebook payloads but are not executable Python code-cell input.
- Malformed notebooks fail closed with a HIGH `NOTEBOOK-MALFORMED` finding because the scanner cannot prove code cells were inspected.

The release package remains unchanged and still contains the same two runtime files:

```text
Scan-PythonPackages.ps1
inspect_binary.py
```

Notebook support is implemented in the PowerShell scanner itself and does not add a new runtime helper file. `inspect_binary.py` is a required file in the v1.6.0 release package for native PE/ELF binary inspection.

### What's New in v1.5.2

v1.5.2 was a documentation and packaging patch release. It did not change scanner analysis behavior from v1.5.1. It corrected the operator-facing release package so native binary inspection is available when the scanner is downloaded from GitHub Releases.

The release package explicitly contains both runtime files:

```text
Scan-PythonPackages.ps1
inspect_binary.py
```

These two files must stay in the same directory. `Scan-PythonPackages.ps1` locates the helper with `Join-Path $PSScriptRoot 'inspect_binary.py'`; if the helper is absent, the scanner logs a warning and skips PE/ELF binary triage.

### What's New in v1.5

v1.5 adds triage-level static inspection for native compiled binaries (`.pyd`, `.so`, `.dll`) using `pefile` and `pyelftools`. The previous `Find-NativeBinaries` function, which only reported the presence of native binaries as a generic `MEDIUM` warning, has been replaced by `Invoke-BinaryInspection`, which performs structural analysis and produces specific, actionable findings. The scanner now examines:

- **Format validity** - whether the file is a valid PE or ELF binary, or is disguised/corrupt.
- **Digital signatures** - whether a PE binary carries a signature (Authenticode certificate directory).
- **Import table analysis** - detection of suspicious APIs (process injection, anti-analysis), sensitive APIs (credential access, DPAPI, registry), and unexpected network/crypto library imports.
- **Section entropy** - Shannon entropy above 7.0 flags possible packing or encryption.
- **Section names** - non-standard section names are flagged for manual review.

v1.5 also migrates the project layout to a git repository with canonical paths (`src/`, `tests/`, `docs/`, `.github/workflows/`). Versioned filenames (`Scan-PythonPackages_v1_3`, `_v1_4`) are replaced by a single `src/Scan-PythonPackages.ps1` with the version tracked via `.NOTES` and git tags.

## Purpose & Use Case

This tool provides a single-command entry point for review operators to:

- Triage incoming Python packages received via media transfer.
- Surface high-risk patterns, hardcoded secrets, and known CVEs prior to approval.
- Inspect native compiled binaries for suspicious imports, packing, and structural anomalies.
- Produce a flat, human-readable report suitable for non-technical reviewers and an audit-grade run log for traceability.
- Inspect Jupyter notebook code cells without executing notebooks or starting kernels.

It is intended to run on a Windows host with PowerShell 5.1+ and Python 3.x available locally.

## Installation Layout

v1.6 retains the same installation layout as v1.5.2. Files are separated by lifecycle and trust boundary, writing to three different filesystem locations during a scan:

### Release package contents

Operators should download the release zip, not the standalone PowerShell file. The zip is intentionally small and contains only the runtime files needed to operate the scanner:

```text
Scan-PythonPackages.ps1
inspect_binary.py
```

Install by extracting both files to the same writable directory, for example:

```text
C:\Tools\Scan-PythonPackages\
  Scan-PythonPackages.ps1
  inspect_binary.py
```

Do not move `inspect_binary.py` to another folder. The scanner resolves it relative to the script directory (`$PSScriptRoot`). If the helper is missing, all non-binary analyzers still run, but native binary inspection is skipped and the run log records `Binary inspection helper not found`.

| Location | Contents | Lifecycle |
|---|---|---|
| `<script-dir>\Scan-PythonPackages.ps1` | Main PowerShell scanner entry point | Shipped in release zip |
| `<script-dir>\inspect_binary.py` | Required Python helper for PE/ELF native binary inspection | Shipped in release zip; must remain beside the script |
| `<script-dir>\.scan-venv\` | Scanner virtual environment with bandit, detect-secrets, pip-audit, pefile, pyelftools | Created on first run, reused across all subsequent scans |
| `<script-dir>\logs\` | Per-run diagnostic log (`run_<timestamp>.log`) | Append-only; never overwritten |
| `%TEMP%\python-scanner-<timestamp>\` | Extraction staging for archives | Created at scan start, **deleted at scan end** |
| `<scan-root>\.reports\` | Operator summary report (`summary_<timestamp>.txt`) | One per scan, never overwritten |

The deliberate split of these locations means:

- **The submission folder stays clean** - extracted (and possibly untrusted) package contents are written to TEMP, not into the operator's submission folder, and are removed at the end of every run.
- **One scanner install, many scans** - the venv lives next to the script, so installing scanner dependencies once is sufficient regardless of how many submission folders the operator processes.
- **Logs are centrally located** - operators reviewing past scans can find every run log in one place (`<script-dir>\logs\`) without needing to remember which submission folders were scanned when.
- **The summary report stays with the submission** - the operator-facing report goes into the scan folder so it travels with the package being reviewed.

The script logs all four paths at startup so they appear in every run log:

```
[INFO] Scan root  : C:\incoming\submission_2026-04-28
[INFO] Report dir : C:\incoming\submission_2026-04-28\.reports
[INFO] Scanner venv: C:\tools\.scan-venv
[INFO] Log file   : C:\tools\logs\run_20260428_140312.log
[INFO] Staging dir: C:\Users\reviewer\AppData\Local\Temp\python-scanner-20260428_140312
```

## Features in v1.6

### Static analysis pipeline
The scanner orchestrates three independent analyzers, a notebook parser/projection pass, and a binary inspection pass, then aggregates their results into a single per-unit risk verdict:

- **Bandit** - Risky Python code patterns (`eval`, `exec`, `pickle`, `subprocess` shell, weak crypto, insecure network calls).
- **detect-secrets** - Hardcoded credentials, tokens, and API keys in Python source.
- **pip-audit** - Known CVEs against declared `Requires-Dist` dependencies, validated against the PyPI Advisory Database.
- **Binary inspection** - Triage-level static analysis of `.pyd`, `.so`, and `.dll` artifacts: format validation, import table inspection, section entropy analysis, and PE signature checks.
- **Notebook parser** - Static `.ipynb` code-cell projection plus explicit warnings for malformed notebooks, saved outputs, and attachments.

### Notebook inspection detail

The notebook support added in v1.6.0 is intentionally implemented as a projection layer in front of the existing static analyzers. The scanner does not execute notebooks, does not invoke Jupyter, and does not start a Python kernel. It treats the `.ipynb` file as JSON input and extracts only code-cell text.

**Loose notebooks.** `Get-PackageUnits` now emits a unit with `Kind = 'notebook'` for each `.ipynb` file discovered under the scan root. The main scan loop converts each notebook to a temporary `.py` file in the existing `%TEMP%\python-scanner-<timestamp>\` workspace and then runs Bandit and detect-secrets against that generated source file.

**Notebooks inside archives.** Archive behavior is additive. After a supported archive is extracted, the scanner searches the extracted directory for `.ipynb` files. Each notebook is projected into a temporary `.py` file under the staging workspace, then the projection directory is scanned with Bandit and detect-secrets. Existing archive analysis still runs against the extracted archive directory, and pip-audit/SBOM/binary inspection behavior is unchanged.

**Projection content.** The generated Python source includes two header comments and one `# Cell <n>` marker before each copied code cell. These markers help reviewers map static-analysis findings back to notebook cell order. Markdown cells, raw cells, metadata, saved outputs, and attachments are not copied into the Python source projection.

**Notebook parser findings.** Non-code payloads and parse failures are reported explicitly:

| Finding | Severity | Meaning |
|---|---|---|
| `NOTEBOOK-MALFORMED` | HIGH | The notebook could not be parsed as valid notebook JSON, or it did not contain a usable `cells` array. Code-cell analysis could not be completed. |
| `NOTEBOOK-SAVED-OUTPUT` | MEDIUM | A notebook cell contains saved output. The output is visible payload data but is not analyzed as Python code. |
| `NOTEBOOK-ATTACHMENT` | MEDIUM | A notebook cell contains one or more attachments. Attachments are visible payload data but are not analyzed as Python code. |

Malformed notebooks fail closed with a HIGH finding because the scanner cannot verify that code cells were inspected. Saved outputs and attachments are MEDIUM findings because they may contain rendered HTML, images, JavaScript, or other payloads that deserve manual review, but they are outside the scanner's Python code-cell analysis scope.

### Binary inspection detail

The `Invoke-BinaryInspection` function discovers `.pyd`, `.so`, and `.dll` files within each package unit and delegates analysis to `src/inspect_binary.py`, a Python helper that uses `pefile` (for PE/Windows binaries) and `pyelftools` (for ELF/Linux binaries).

In the source repository the helper lives at `src/inspect_binary.py`. In the operator release package it lives directly beside `Scan-PythonPackages.ps1`. Both layouts are valid because the scanner uses `$PSScriptRoot` to find the helper next to the running script.

**Detection by extension and magic bytes.** Files are first routed by extension (`.pyd`/`.dll` -> PE, `.so` -> ELF). If the extension is ambiguous, the 4-byte file magic is checked (`MZ` for PE, `\x7fELF` for ELF). Files that claim a native extension but fail both format parsers are flagged as `BINARY-INVALID-FORMAT` at HIGH severity - this catches disguised or corrupt binaries that could be used to smuggle arbitrary data.

**PE analysis** covers:

| Check | Finding | Severity |
|---|---|---|
| Invalid PE format | `BINARY-INVALID-FORMAT` | HIGH |
| Process injection APIs (`CreateRemoteThread`, `VirtualAllocEx`, `WriteProcessMemory`, `NtCreateThreadEx`, etc.) | `BINARY-SUSPICIOUS-IMPORT` | HIGH |
| Anti-analysis APIs (`IsDebuggerPresent`, `CheckRemoteDebuggerPresent`, `NtQueryInformationProcess`) | `BINARY-SUSPICIOUS-IMPORT` | HIGH |
| Credential/registry access (`CredRead`, `CryptUnprotectData`, `RegOpenKeyEx`) | `BINARY-SENSITIVE-IMPORT` | MEDIUM |
| Network library imports (`ws2_32.dll`, `winhttp.dll`, `wininet.dll`) | `BINARY-NETWORK-IMPORT` | MEDIUM |
| Section entropy > 7.0 | `BINARY-HIGH-ENTROPY` | MEDIUM |
| Non-standard section names | `BINARY-UNUSUAL-SECTION` | LOW |

**ELF analysis** covers:

| Check | Finding | Severity |
|---|---|---|
| Invalid ELF format | `BINARY-INVALID-FORMAT` | HIGH |
| Network/crypto shared library imports (`libcurl`, `libssl`) | `BINARY-NETWORK-IMPORT` | MEDIUM |
| Section entropy > 7.0 | `BINARY-HIGH-ENTROPY` | MEDIUM |
| Non-standard section names | `BINARY-UNUSUAL-SECTION` | LOW |
| Stripped symbol table | `BINARY-STRIPPED-SYMBOLS` | LOW |

**Digital signature detection** is reported in the JSON output (`"signed": true/false`) for PE binaries. A missing signature is informational - it is not elevated to a finding because many legitimate Python extensions are unsigned.

**Output isolation.** The Python helper writes its JSON result to a temp file rather than stdout, matching the pattern used by Bandit and pip-audit. This prevents stderr pollution (deprecation warnings, import warnings) from corrupting the JSON parse. The temp file is cleaned up in a `finally` block.

**Graceful degradation.** If `pefile` or `pyelftools` is not installed, if the Python helper script is missing, or if the venv Python is unavailable, the function logs a `WARN` and returns zero findings rather than aborting the scan.

### Supported input formats
Operators may point the scanner at a folder containing any combination of:

- Archives: `.whl`, `.egg`, `.zip`, `.tar.gz`, `.tgz`
- Loose Python source: `.py`, `.pyw`
- Jupyter notebooks: `.ipynb`

Compound suffixes (`.tar.gz`) are matched correctly via filename suffix comparison rather than `.Extension`, which only returns the trailing segment.

### Unsupported file detection
`Find-UnsupportedFiles` recursively walks the scan root and identifies files whose suffix is outside the supported archive, Python source, and notebook input sets. `.ipynb` files are no longer reported as unsupported because they are now scanned through code-cell projection. Files under the scan root's `.reports\` directory are excluded so reports from prior runs are not flagged on a re-scan.

### Isolated scanner environment
The scanner provisions a single Python virtual environment at `<script-dir>\.scan-venv\` on first run and reuses it for every subsequent scan. v1.5 adds `pefile` and `pyelftools` to the scanner dependency set alongside `bandit`, `detect-secrets`, and `pip-audit`.

### Dependency management
- Configurable minimum-version floors are enforced for each scanner tool:

| Package | Minimum Version |
|---|---|
| `bandit` | `>= 1.7.0` |
| `pip-audit` | `>= 2.0.0` |
| `detect-secrets` | `>= 1.4.0` |
| `pefile` | `>= 2023.2.7` |
| `pyelftools` | `>= 0.29` |

- Version comparison is **PEP 440-compliant**: `Compare-Versions` shells out to the venv Python and uses `packaging.version.Version`, so pre-releases, post-releases, dev releases, local-version identifiers, and epochs are all ordered correctly.
- The PEP 440 helper tries the standalone `packaging` module first and falls back to `pip._vendor.packaging.version`.
- The check **fails closed**: unparseable versions, a missing `packaging` module, or an unreachable Python executable return `false` and trigger a reinstall rather than silently passing.
- **Install verification does not trust pip's exit code.** After every install the script re-queries the installed version via `pip show` and re-runs `Compare-Versions`; only a passing version check is treated as success.
- Existing scanner installations are reused when present and at-or-above the minimum version.
- `-AutoInstall` switch suppresses interactive prompts for unattended/scheduled execution.

### Archive extraction
- Wheels, eggs, and zip archives are expanded with the built-in `Expand-Archive`. Because PowerShell 5.1's `Expand-Archive` only accepts inputs with a `.zip` extension, `.whl` and `.egg` files are first copied to a temp `.zip` filename, expanded, and then the temp file is removed in a `finally` block.
- Tarballs use system `tar` when present (Windows 10 1803+) and fall back to a Python `tarfile` helper script otherwise.
- All extractions land in `%TEMP%\python-scanner-<timestamp>\unit_<sanitized-name>\` and the entire timestamped staging directory is removed at the end of the run.

### Reporting
Each scan run produces up to four artifacts across two distinct locations:

- `<scan-root>\.reports\summary_<timestamp>.txt` - Operator-facing flat-text report with overall risk verdict (`HIGH` / `MEDIUM` / `LOW` / `CLEAN`), optional unsupported-files warning block, per-unit findings, severity counts, tool coverage notes, and a SBOM status line.
- `<scan-root>\.reports\summary_<timestamp>.json` - Machine-readable JSON report containing the same scan data in a stable, downstream-consumable schema (see *JSON Report Schema* below).
- `<scan-root>\.reports\sbom_<timestamp>_<unit>.cdx.json` - CycloneDX 1.4 SBOM (one per archive with declared `Requires-Dist` metadata).
- `<script-dir>\logs\run_<timestamp>.log` - Full diagnostic log with `INFO`/`WARN`/`ERROR`/`DEBUG` entries for audit and troubleshooting.

The tool coverage block in the operator report now includes binary inspection and notebook parsing:

```
  TOOL COVERAGE
================================================================

  Bandit          : Code pattern analysis (eval, pickle, subprocess, weak crypto)
  detect-secrets  : Hardcoded secrets, tokens, API keys, credentials
  pip-audit       : Dependency CVE check against PyPI Advisory Database
  BinaryInspect   : Native binary triage (format validation, imports, entropy, signatures)
  NotebookParser  : Jupyter notebook code-cell projection and notebook payload warnings

  NOTE: This report is based on STATIC analysis only.
        No package code was executed during this scan.
        Native binaries cannot be fully assessed by static analysis.
```

### JSON Report Schema

The JSON report schema remains `schemaVersion: "1.0"`. v1.6.0 adds notebook-related values without changing the shape of the report. Findings from binary inspection use `Tool: "BinaryInspection"` and carry test IDs prefixed with `BINARY-`. Findings from notebook parsing use `Tool: "NotebookParser"` and carry test IDs prefixed with `NOTEBOOK-`. For notebook parser findings, `Line` is the notebook cell number when the finding is cell-specific and `0` when the whole notebook could not be parsed.

```json

  "scannerVersion": "1.6.0",
  "scanDate": "<ISO 8601 UTC>",

  "unsupportedFiles": [{ "relativePath": "...", "extension": "...", "sizeBytes": 0 }],

    "name": "example-1.0.whl",
    "kind": "archive | pyfile | notebook",
    "risk": "HIGH | MEDIUM | LOW | CLEAN",

      "Tool": "Bandit | detect-secrets | pip-audit | BinaryInspection | NotebookParser",

### CycloneDX SBOM
Unchanged from v1.4. pip-audit natively supports `--format cyclonedx-json`; one SBOM is produced per archive that declares at least one `Requires-Dist` dependency.

### Operator UX
- Color-coded console output separates progress status from structured log entries.
- Banner and completion screens clearly demarcate run boundaries.
- Final report path is echoed on completion for direct copy-paste.

### PowerShell 5.1 robustness
The target deployment runtime is Windows PowerShell 5.1. v1.6 retains all robustness patterns from v1.5.2 and applies them consistently to notebook parsing and projection:

- **Stderr capture without false termination.** `Invoke-BinaryInspection` uses the same `$prevEAP = $ErrorActionPreference; $ErrorActionPreference = 'Continue'; ...; $ErrorActionPreference = $prevEAP` block on the very next line after the native call (matching the Bandit/pip-audit pattern).
- **JSON-on-stdout protected from stderr pollution.** The Python helper writes JSON to a temp file rather than stdout, so stderr warnings from `pefile` or `pyelftools` cannot corrupt the JSON parse.
- **Temp file cleanup in `finally` block.** The binary inspection temp JSON file is removed in a `finally` block, matching the cleanup pattern used by all other temp files in the script.
- All other v1.4 robustness measures (defensive detect-secrets parsing, `@(...)` array wrapping, explicit source-file counters, deterministic finding aggregation, `Expand-Archive` extension workaround) remain unchanged.

```

## Logs & Troubleshooting

### Where logs are written

Every run produces a timestamped log next to the script:

```
<script-dir>\logs\run_<yyyymmdd_hhmmss>.log
```

### What the log contains

Entries are formatted as `[yyyy-mm-dd HH:mm:ss] [LEVEL] message` with four levels:

- **INFO** - Normal lifecycle events (Python found, venv created, scanner versions, per-tool finding counts, binary inspection results).
- **WARN** - Recoverable issues (tool missing, extraction skipped, scanner errored but run continues, binary inspection skipped).
- **ERROR** - Fatal conditions that abort the run (Python not found, venv corrupt, dependency install failed).
- **DEBUG** - Verbose tool output captured from `pip`, `bandit`, `pip-audit`, `detect-secrets`, `inspect_binary.py`, and tarball extraction.

### Common troubleshooting flows

| Symptom | Where to look | Likely cause |
|---|---|---|
| Operator report shows fewer findings than expected | Search log for `WARN.*not available` | Scanner tool missing from venv - re-run with `-AutoInstall` or stage wheels offline |
| Binary inspection skipped | Search log for `Binary inspection Python not available` or `Binary inspection helper not found` | `pefile`/`pyelftools` not installed, or `inspect_binary.py` not in `src/` next to the script |
| Binary inspection produced no output | Search log for `produced no output file` | Python helper crashed or timed out - check `DEBUG binary-inspect:` lines for stderr |
| Run aborts immediately | First `ERROR` line in log | Python not on `PATH`, or venv `pip.exe` missing |
| `pip-audit` reports 0 findings on a wheel | Search for `No Requires-Dist metadata found` | Wheel has no `METADATA` file - CVE audit cannot run |
| SBOM not produced for a wheel | Search for `No Requires-Dist metadata found` | Same root cause - no declared dependencies means no SBOM |
| Tarball extraction fails | Search for `Extraction failed` and surrounding `DEBUG` | System `tar` missing and Python tarfile fallback also failed |
| Notebook reports `NOTEBOOK-MALFORMED` | Review the notebook file and log line `Notebook parse error` | Invalid notebook JSON or missing `cells` array; scanner could not inspect code cells |
| Notebook has `NOTEBOOK-SAVED-OUTPUT` or `NOTEBOOK-ATTACHMENT` | Review the notebook manually | Visible non-code notebook payloads were present but not analyzed as Python code |

## Offline / Manual Installation of Scanner Tools

If the host running the scanner cannot reach PyPI (air-gapped network, proxy block, intermittent outage), the scanner tools must be staged manually. v1.6 does not add any new Python package dependencies beyond the v1.5 binary inspection set.

### 1. Download wheels on a network-connected machine

```powershell
mkdir C:\offline-scanner-wheels
<python> -m pip download bandit         -d C:\offline-scanner-wheels
<python> -m pip download detect-secrets -d C:\offline-scanner-wheels
<python> -m pip download pip-audit      -d C:\offline-scanner-wheels
<python> -m pip download pefile         -d C:\offline-scanner-wheels
<python> -m pip download pyelftools     -d C:\offline-scanner-wheels
```

Match the Python version, OS, and CPU architecture of the target scanner host. Both `pefile` and `pyelftools` are pure-Python packages with no native dependencies, so cross-platform wheel compatibility is not a concern for these two.

### 2. Transfer and install

Transfer the wheels folder to the scanner host and install into the scanner venv:

```powershell
$venvPip = "<script-dir>\.scan-venv\Scripts\pip.exe"

& $venvPip install --no-index --find-links C:\offline-scanner-wheels bandit detect-secrets pip-audit pefile pyelftools
```

### 3. Verify

```powershell
& $venvPip show bandit detect-secrets pip-audit pefile pyelftools
```

## Testing & Validation

### Automated Tests

A Pester v5 test suite lives in `tests/Scan-PythonPackages.Tests.ps1`.

**How to run:**
```powershell
.\tests\Run-Tests.ps1
```

**Prerequisites:** Pester 5.x (`Install-Module Pester -MinimumVersion 5.0 -Scope CurrentUser`); Python 3 on PATH for the `Compare-Versions` integration tests and `Invoke-BinaryInspection` tests.

**Coverage:**

| Describe block | Type | What is covered |
|---|---|---|
| `Compare-Versions` | Integration | 17 cases: exact match, semver above/below, PEP 440 pre-releases, post-releases, local-version, epochs, empty inputs, garbage strings, missing executable |

## Known Limitations

- **Static-only analysis.** No dynamic execution, sandbox detonation, or runtime instrumentation is performed. Packages that obfuscate behavior at runtime will not be flagged.
- **Binary inspection is triage-level.** Import table analysis, section entropy, and format validation provide useful signals, but cannot detect all malicious behavior in native code. Packed or obfuscated binaries may evade static analysis. Findings should be treated as triage indicators for further manual or dynamic analysis.
- **PE signature detection, not verification.** The scanner reports whether a PE binary carries a signature (Authenticode certificate directory), but does not verify the signature chain, check revocation, or validate the signing certificate. A binary reported as "signed" may have an invalid, expired, or self-signed certificate.
- **CVE coverage depends on declared metadata.** `pip-audit` reads `Requires-Dist` from each archive's `METADATA`. Packages without metadata cannot be CVE-audited.
- **Notebook scope is code-cell projection.** Markdown cells are not analyzed as Python. Saved outputs and attachments are reported for manual review but are not deeply inspected.
- **Windows-only paths.** Venv layout and binary extension lookups assume a Windows host. Cross-platform support is not in scope for v1.6.
- **First-run network dependency.** Scanner installation requires outbound connectivity to PyPI. Subsequent runs are offline-capable, but `pip-audit` continues to query the PyPI Advisory Database on every run unless an offline feed is staged.
- **Script directory must be writable.** v1.6 writes the venv and run logs next to the script. If the script is installed on read-only media, scans will fail at venv creation.

## Backwards Compatibility

**Upgrading from v1.5.2 -> v1.6.0** does not require a dependency rebuild. Replace the v1.5.2 scanner files with the v1.6.0 release zip and keep `Scan-PythonPackages.ps1` and `inspect_binary.py` together in the extracted directory. Notebook support is implemented in the PowerShell script and does not add a new Python package dependency.

**Upgrading from v1.5.1 -> v1.5.2** did not require a dependency rebuild. The v1.5.2 release corrected the release package so the script and binary helper are distributed together.

**Upgrading from v1.4 -> v1.5** requires a venv rebuild to install the new `pefile` and `pyelftools` dependencies. Delete `<script-dir>\.scan-venv\` and re-run with `-AutoInstall`, or install the two new packages manually:

```powershell
& "<script-dir>\.scan-venv\Scripts\pip.exe" install pefile pyelftools
```

The JSON report schema (`schemaVersion: "1.0"`) is unchanged. v1.6.0 adds `notebook` as a possible unit kind and `NotebookParser` as a possible finding tool. Downstream consumers that do not recognize notebook findings can ignore them without breaking schema parsing. The `NativeBinaryCheck` tool name from v1.4 is no longer emitted.

The operator `.txt` report includes `BinaryInspect` and `NotebookParser` lines in the tool coverage block and may include additional finding lines for notebook parser warnings. The overall structure is unchanged.

## Security & Operational Notes

- The scanner never executes submitted package code or notebook code. All analysis is static.
- `inspect_binary.py` uses `pefile` and `pyelftools` to parse binary structures. Neither library executes the analyzed binary.
- Binary inspection JSON output is written to a temp file in `%TEMP%` and cleaned up in a `finally` block, matching the pattern used by all other temp files in the script.
- Extracted package contents and notebook projections do not linger in the submission folder. Archive extraction and notebook `.py` projections happen under `%TEMP%\python-scanner-<timestamp>\`, and the entire directory is removed at the end of every successful run.
- The scanner venv is isolated from the system Python install. Removing `<script-dir>\.scan-venv\` is sufficient to reset scanner state.
- `Set-StrictMode -Version Latest` and `$ErrorActionPreference = 'Stop'` are enabled at module scope.
