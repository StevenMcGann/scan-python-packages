## Overview

v1.3 of `Scan-PythonPackages.ps1`, a PowerShell-based static security scanner for Python packages submitted through the media transfer review workflow. The tool is designed for operator-driven inspection of untrusted Python artifacts before they are admitted to a trusted environment.

All analysis performed by this tool is **strictly static** — no submitted package code is executed at any point during the scan.

> **Note on version history.** The script went through several internal drafts (`1.0`, `1.0-PEP440`) before the v1.2 GA release, which established the stable installation layout (venv, logs, staging, and reports all in separate locations). v1.3 builds on v1.2 by adding the Pester test harness (`tests/`). The installation layout, output paths, and lifecycle semantics are unchanged from v1.2.

---

## Purpose & Use Case

This tool provides a single-command entry point for review operators to:

- Triage incoming Python packages received via media transfer.
- Surface high-risk patterns, hardcoded secrets, and known CVEs prior to approval.
- Produce a flat, human-readable report suitable for non-technical reviewers and an audit-grade run log for traceability.

It is intended to run on a Windows host with PowerShell 5.1+ and Python 3.x available locally.

## Installation Layout

v1.3 separates files by lifecycle and trust boundary, writing to three different filesystem locations during a scan:

| Location | Contents | Lifecycle |
|---|---|---|
| `<script-dir>\.scan-venv\` | Scanner virtual environment with bandit, detect-secrets, pip-audit | Created on first run, reused across all subsequent scans |
| `<script-dir>\logs\` | Per-run diagnostic log (`run_<timestamp>.log`) | Append-only; never overwritten |
| `%TEMP%\python-scanner-<timestamp>\` | Extraction staging for archives | Created at scan start, **deleted at scan end** |
| `<scan-root>\.reports\` | Operator summary report (`summary_<timestamp>.txt`) | One per scan, never overwritten |

The deliberate split of these locations means:

- **The submission folder stays clean** — extracted (and possibly untrusted) package contents are written to TEMP, not into the operator's submission folder, and are removed at the end of every run.
- **One scanner install, many scans** — the venv lives next to the script, so installing scanner dependencies once is sufficient regardless of how many submission folders the operator processes.
- **Logs are centrally located** — operators reviewing past scans can find every run log in one place (`<script-dir>\logs\`) without needing to remember which submission folders were scanned when.
- **The summary report stays with the submission** — the operator-facing report goes into the scan folder so it travels with the package being reviewed.

The script logs all four paths at startup so they appear in every run log:

```
[INFO] Scan root  : C:\incoming\submission_2026-04-28
[INFO] Report dir : C:\incoming\submission_2026-04-28\.reports
[INFO] Scanner venv: C:\tools\.scan-venv
[INFO] Log file   : C:\tools\logs\run_20260428_140312.log
[INFO] Staging dir: C:\Users\reviewer\AppData\Local\Temp\python-scanner-20260428_140312
```

## Features in v1.3

### Static analysis pipeline
The scanner orchestrates three independent analyzers plus a native binary check, and aggregates their results into a single per-unit risk verdict:

- **Bandit** — Risky Python code patterns (`eval`, `exec`, `pickle`, `subprocess` shell, weak crypto, insecure network calls).
- **detect-secrets** — Hardcoded credentials, tokens, and API keys in Python source.
- **pip-audit** — Known CVEs against declared `Requires-Dist` dependencies, validated against the PyPI Advisory Database.
- **Native binary check** — Detection of `.pyd`, `.so`, and `.dll` artifacts that fall outside the reach of static Python analyzers.

### Supported input formats
Operators may point the scanner at a folder containing any combination of:

- Archives: `.whl`, `.egg`, `.zip`, `.tar.gz`, `.tgz`
- Loose Python source: `.py`, `.pyw`

Compound suffixes (`.tar.gz`) are matched correctly via filename suffix comparison rather than `.Extension`, which only returns the trailing segment. Because v1.3 keeps the venv outside the scan root and stages extractions in TEMP, the previously-required workspace-exclusion filter inside `Get-PackageUnits` has been removed — there is nothing in the scan root that needs to be skipped.

### Isolated scanner environment
The scanner provisions a single Python virtual environment at `<script-dir>\.scan-venv\` on first run and reuses it for every subsequent scan. Tools are installed once, then shared across all submissions. To force a clean reinstall, delete the `.scan-venv\` folder.

### Dependency management
- Configurable minimum-version floors are enforced for each scanner (`bandit ≥ 1.7.0`, `pip-audit ≥ 2.0.0`, `detect-secrets ≥ 1.4.0`).
- Version comparison is **PEP 440-compliant**: `Compare-Versions` shells out to the venv Python and uses `packaging.version.Version`, so pre-releases (`1.7.0rc1`), post-releases (`.post1`), dev releases (`.dev0`), local-version identifiers (`+local.1`), and epochs (`2!1.0.0`) are all ordered correctly.
- The PEP 440 helper tries the standalone `packaging` module first and falls back to `pip._vendor.packaging.version` — which is bundled inside `pip` itself — so the check works the moment a venv is created, before the first scanner install completes.
- The check **fails closed**: unparseable versions, a missing `packaging` module, or an unreachable Python executable return `false` and trigger a reinstall rather than silently passing.
- **Install verification does not trust pip's exit code.** After every install the script re-queries the installed version via `pip show` and re-runs `Compare-Versions`; only a passing version check is treated as success. This catches cases where pip exits 0 but the requested version was not actually installed (or fell below the minimum-version floor).
- Existing scanner installations are reused when present and at-or-above the minimum version.
- `-AutoInstall` switch suppresses interactive prompts for unattended/scheduled execution.

### Archive extraction
- Wheels, eggs, and zip archives are expanded with the built-in `Expand-Archive`. Because PowerShell 5.1's `Expand-Archive` only accepts inputs with a `.zip` extension, `.whl` and `.egg` files are first copied to a temp `.zip` filename, expanded, and then the temp file is removed in a `finally` block.
- Tarballs use system `tar` when present (Windows 10 1803+) and fall back to a Python `tarfile` helper script otherwise.
- All extractions land in `%TEMP%\python-scanner-<timestamp>\unit_<sanitized-name>\` and the entire timestamped staging directory is removed at the end of the run (Step 8 of main execution).

### Reporting
Each scan run produces up to four artifacts across two distinct locations:

- `<scan-root>\.reports\summary_<timestamp>.txt` — Operator-facing flat-text report with overall risk verdict (`HIGH` / `MEDIUM` / `LOW` / `CLEAN`), per-unit findings, severity counts, tool coverage notes, and a SBOM status line. Travels with the submission.
- `<scan-root>\.reports\summary_<timestamp>.json` — Machine-readable JSON report containing the same scan data in a stable, downstream-consumable schema (see *JSON Report Schema* below). Written alongside the `.txt` with a matching timestamp. Both files travel with the submission.
- `<scan-root>\.reports\sbom_<timestamp>_<unit>.cdx.json` — CycloneDX 1.4 SBOM (one per archive with declared `Requires-Dist` metadata). Produced as a side-effect of the existing pip-audit pass; see *CycloneDX SBOM* below.
- `<script-dir>\logs\run_<timestamp>.log` — Full diagnostic log with `INFO`/`WARN`/`ERROR`/`DEBUG` entries for audit and troubleshooting. Stays with the scanner install.

The `.json` and `.cdx.json` paths are each logged at `INFO` level after a successful write (same pattern as the `.txt` log message). `Show-Done` on the console continues to point at the `.txt` only.

The summary report's recommendation language is risk-graded:

- **HIGH** — "DO NOT APPROVE without expert security review. Escalate immediately."
- **MEDIUM** — "Review findings below before approving."
- **LOW** — "Review findings below. Escalate if uncertain."
- **CLEAN** — "No security findings detected."

### JSON Report Schema

The JSON report is written by `Write-JsonReport` to `<scan-root>\.reports\summary_<timestamp>.json` alongside the `.txt` report. It is BOM-free UTF-8. Top-level shape (schema version `1.0`):

```json
{
  "schemaVersion": "1.0",
  "scannerVersion": "1.3",
  "scanDate": "<ISO 8601 UTC, e.g. 2026-04-29T10:49:22.7112938Z>",
  "scanRoot": "<absolute path to scanned folder>",
  "elapsedSeconds": 4.2,
  "overallRisk": "HIGH | MEDIUM | LOW | CLEAN",
  "totals": {
    "units":    3,
    "findings": 5,
    "high":     2,
    "medium":   1,
    "low":      2
  },
  "units": [
    {
      "name": "example-1.0.whl",
      "kind": "archive | pyfile",
      "risk": "HIGH | MEDIUM | LOW | CLEAN",
      "findings": [
        {
          "Tool":       "Bandit | detect-secrets | pip-audit | NativeBinaryCheck",
          "Severity":   "HIGH | MEDIUM | LOW",
          "Confidence": "HIGH | MEDIUM | LOW",
          "File":       "<path or dependency string>",
          "Line":       42,
          "Issue":      "<human-readable description>",
          "TestID":     "<tool-specific identifier>"
        }
      ]
    }
  ]
}
```

`findings` is an empty array (`[]`) for clean units. `Line` is `0` for findings that are not file-position based (native binary check, pip-audit CVEs). `scanDate` is always UTC with the `Z` suffix. `sbomFiles` is an array of absolute paths to any `.cdx.json` files written during the run; it is an empty array `[]` for pyfile-only scans or archives with no `Requires-Dist` metadata.

### CycloneDX SBOM

pip-audit natively supports `--format cyclonedx-json`, so no additional packages are required. The SBOM pass runs as a second, independent pip-audit invocation inside `Invoke-PipAuditScan` immediately after the CVE pass, reusing the same temp requirements file — the dependency set is therefore identical for both passes. Both pip-audit invocations use `--no-deps --disable-pip` because the scanner is auditing static `Requires-Dist` declarations exactly as submitted, not resolving or installing a dependency graph.

**Output:** `<scan-root>\.reports\sbom_<timestamp>_<unit>.cdx.json` — one file per archive that declares at least one `Requires-Dist` dependency. The `<unit>` segment is the archive filename with its extension stripped and special characters replaced by `_`. Loose `.py` / `.pyw` files and archives without `METADATA` produce no SBOM.

**Choice of one-per-archive vs. merged:** per-archive was chosen because each archive's dependency set is independent and corresponds directly to a single `Invoke-PipAuditScan` call. Merging would require an extra post-processing step with no practical benefit for the media-transfer review workflow.

**SBOM failure is non-fatal.** If the SBOM pass fails (network issue, pip-audit error), a `WARN` is logged and the CVE findings are returned normally. The text and JSON summary reports record `"SBOM: not produced"` in the header when no SBOM is written.

**Format details.** The CycloneDX output is written directly by pip-audit as UTF-8 (no BOM). The file is spec version `1.4` and includes `bomFormat`, `specVersion`, `serialNumber`, `version`, `metadata`, and `components` arrays with each dependency's name, version, PURL, and any known vulnerability records.

### Operator UX
- Color-coded console output separates progress status (`Show-Status`) from structured log entries (`Write-Log`).
- Banner and completion screens clearly demarcate run boundaries.
- Final report path is echoed on completion for direct copy-paste.

### PowerShell 5.1 robustness
The target deployment runtime is Windows PowerShell 5.1, which has several behaviors that bite naive native-command invocations. v1.3 includes a robustness pass for each of them:

- **Stderr capture without false termination.** With `$ErrorActionPreference = 'Stop'`, native commands that write to stderr (Python, pip, bandit, pip-audit, tar) raise `NativeCommandError` when their output is captured via `2>&1`, even on a successful exit. The script wraps every native-command invocation — including both branches of the tarball extractor — in a `$prevEAP = $ErrorActionPreference; $ErrorActionPreference = 'Continue'; ...; $ErrorActionPreference = $prevEAP` block so stderr-as-progress doesn't kill the run, while strict-mode protection is restored immediately afterwards.
- **JSON-on-stdout protected from stderr pollution.** `detect-secrets` writes its findings to stdout as JSON. Capturing its stderr with `2>&1` and piping the merged stream to a file produces unparseable output. The script redirects detect-secrets' stderr to `$null` instead, keeping the JSON clean. The scan runs with `--all-files --no-verify` from the target directory so extracted archives and loose files are scanned as static local content rather than depending on git-tracked-file discovery or network-backed credential verification.
- **Defensive parsing of detect-secrets output.** Strict mode treats access to a non-existent property on a `PSCustomObject` as a hard error, so the parser checks `$raw.PSObject.Properties['results']` before reading `$raw.results`, gracefully handling versions of detect-secrets (or empty scans) that omit the key.
- **Pipeline-result counting.** PowerShell unwraps single-element pipelines into scalars, which means `(... | Where-Object { ... }).Count` can return `$null` instead of `0`. All `.Count` reads on pipeline results are wrapped in `@(...)` to force array semantics — affecting `Get-RiskLevel`, the severity-count rollup, the per-unit `FINDINGS:` line, the deduplicated `Requires-Dist` list, the discovery loops in `Get-PackageUnits`, and the `$units` assignment in the main block (`$units = @(Get-PackageUnits ...)` guards against the single-unit unwrap that would otherwise surface under strict mode).
- **Explicit source-file counter preserved.** `Get-PackageUnits` increments a dedicated `$pyFilesAdded` counter inside the discovery loop. In v1.3 every discovered `.py`/`.pyw` file is added (the v1.0 workspace-exclusion filter is no longer needed because the venv and staging both live outside the scan root), so this counter currently equals `$pyFiles.Count`. The explicit counter is retained to guard against silent miscounting if filtering is ever reintroduced.
- **Deterministic finding aggregation.** The main loop appends per-tool findings via an explicit `foreach { $list.Add($f) }` instead of `List.AddRange(@(...))`, which is more tolerant of empty/scalar pipeline outputs that would otherwise raise `ArgumentNullException` against the typed list.
- **`Expand-Archive` extension workaround.** See the *Archive extraction* subsection above.

## Logs & Troubleshooting

### Where logs are written

Every run produces a timestamped log next to the script:

```
<script-dir>\logs\run_<yyyymmdd_hhmmss>.log
```

The path is also echoed at the top of the run via `Write-Log` (`Log file : ...`). The log is UTF-8 encoded, append-only within a run, and never overwrites prior runs — each invocation gets its own timestamped file. Because logs accumulate in a single directory across all scans, operators can browse run history without needing to remember which submission folders were scanned when.

### What the log contains

Entries are formatted as `[yyyy-mm-dd HH:mm:ss] [LEVEL] message` with four levels:

- **INFO** — Normal lifecycle events (Python found, venv created, scanner versions, per-tool finding counts, units processed).
- **WARN** — Recoverable issues (tool missing, extraction skipped, scanner errored but run continues, native binaries detected).
- **ERROR** — Fatal conditions that abort the run (Python not found, venv corrupt, dependency install failed).
- **DEBUG** — Verbose tool output captured from `pip`, `bandit`, `pip-audit`, `detect-secrets`, and tarball extraction.

### Common troubleshooting flows

| Symptom | Where to look | Likely cause |
|---|---|---|
| Operator report shows fewer findings than expected | Search log for `WARN.*not available` | Scanner tool missing from venv — re-run with `-AutoInstall` or stage wheels offline |
| Run aborts immediately | First `ERROR` line in log | Python not on `PATH`, or venv `pip.exe` missing |
| `pip-audit` reports 0 findings on a wheel | Search for `No Requires-Dist metadata found` | Wheel has no `METADATA` file — CVE audit cannot run |
| SBOM not produced for a wheel | Search for `No Requires-Dist metadata found` | Same root cause as above — no declared dependencies means no SBOM. Expected for pure CA-bundle or data-only wheels (e.g. `certifi`) |
| Tarball extraction fails | Search for `Extraction failed` and surrounding `DEBUG` | System `tar` missing and Python tarfile fallback also failed |
| Run hangs during dependency install | Tail the log; pip install output is verbose, so the most recent `[DEBUG]` lines reflect what pip is currently doing | Network stall reaching PyPI |
| Need to confirm scanner versions actually used | Search for `is installed (version` | Each scanner logs its resolved version at startup |
| Staging directory left behind in TEMP | Run aborted before Step 8 | Inspect the log's last entry; the directory can be safely deleted manually |
| Need to find logs for a past scan | `<script-dir>\logs\` | All run logs live there in one place, regardless of which scan folder was used |

### Tailing a run live

The log is written incrementally, so a second PowerShell window can follow it during a long scan:

```powershell
Get-Content -Path "<script-dir>\logs\run_<timestamp>.log" -Wait -Tail 20
```

### Sharing logs for escalation

When escalating a finding or reporting a scanner bug, attach **both** files:

- `<scan-root>\.reports\summary_<timestamp>.txt` — what was found.
- `<script-dir>\logs\run_<timestamp>.log` — how it was found, which tools ran, and any warnings encountered.

The log contains no submitted package contents — only filenames, scanner output, and tool diagnostics — so it is safe to share with reviewers who are not cleared to see the package itself.

## Offline / Manual Installation of Scanner Tools

If the host running the scanner cannot reach PyPI (air-gapped network, proxy block, intermittent outage), the scanner tools must be staged manually. The scanner will print a `WARN` and skip the corresponding scan stage if a tool is missing from the venv `Scripts` directory, so operators should pre-stage them before the run.

The same procedure applies to all three scanner tools: **bandit**, **detect-secrets**, and **pip-audit**.

### 1. Download wheels on a network-connected machine

From any host with PyPI access, use `pip download` to fetch each tool **and all of its transitive dependencies** as wheels into a single folder:

```powershell
# Replace <python> with py / python / python3 as appropriate
mkdir C:\offline-scanner-wheels
<python> -m pip download bandit         -d C:\offline-scanner-wheels
<python> -m pip download detect-secrets -d C:\offline-scanner-wheels
<python> -m pip download pip-audit      -d C:\offline-scanner-wheels
```

Match the Python version, OS, and CPU architecture of the **target** scanner host:

```powershell
<python> -m pip download pip-audit `
    --python-version 3.11 `
    --platform win_amd64 `
    --only-binary=:all: `
    -d C:\offline-scanner-wheels
```

If a pure-source dependency is required and no matching wheel exists, drop `--only-binary=:all:` to allow `.tar.gz` source distributions; the target host will need a build toolchain in that case.

Alternatively, download the wheel for a specific version directly from PyPI:

- `https://pypi.org/project/bandit/#files`
- `https://pypi.org/project/detect-secrets/#files`
- `https://pypi.org/project/pip-audit/#files`

### 2. Transfer wheels to the scanner host

Copy the entire `offline-scanner-wheels` folder to the scanner host via the same media transfer channel used for package submissions. Keep all dependency wheels together — `pip` resolves them as a set.

### 3. Install into the scanner venv

The scanner venv lives at `<script-dir>\.scan-venv` (i.e., next to `Scan-PythonPackages_v1_3.ps1`). Activate the venv's `pip` directly and install from the local folder using `--no-index` (forces fully offline resolution):

```powershell
$venvPip = "<script-dir>\.scan-venv\Scripts\pip.exe"

& $venvPip install --no-index --find-links C:\offline-scanner-wheels bandit
& $venvPip install --no-index --find-links C:\offline-scanner-wheels detect-secrets
& $venvPip install --no-index --find-links C:\offline-scanner-wheels pip-audit
```

To upgrade existing installs to a newer version staged in the same folder:

```powershell
& $venvPip install --no-index --find-links C:\offline-scanner-wheels --upgrade bandit detect-secrets pip-audit
```

Verify the install:

```powershell
& $venvPip show bandit
& $venvPip show detect-secrets
& $venvPip show pip-audit
```

The scanner reuses the same venv across runs and across submissions, so once staged the tools persist until the venv is deleted.

### 4. Stage the pip-audit vulnerability database

`pip-audit` queries the PyPI Advisory Database (OSV) over the network on every run. For air-gapped operation, download the OSV vulnerability feed on a connected host and point `pip-audit` at the local copy via its `--cache-dir` and `--vulnerability-service osv` options. Refresh the cache on a defined cadence (weekly minimum) so CVE coverage does not drift.

A separate offline-feed sync process is recommended for sustained air-gapped use; v1.3 of the scanner does not automate this.

### 5. Verifying offline readiness before a scan

Run the scanner once with `-AutoInstall` after staging. Confirm the run log contains:

```
[INFO] bandit is installed (version X.Y.Z) — OK.
[INFO] detect-secrets is installed (version X.Y.Z) — OK.
[INFO] pip-audit is installed (version X.Y.Z) — OK.
```

If any line reads `WARN` or the install attempts to reach PyPI, the wheels were not picked up — verify `--find-links` path, wheel filenames, and Python version compatibility.

## Known Limitations

- **Static-only analysis.** No dynamic execution, sandbox detonation, or runtime instrumentation is performed. Packages that obfuscate behavior at runtime will not be flagged.
- **Native binaries are reported but not analyzed.** `.pyd`, `.so`, and `.dll` files are flagged at `MEDIUM` severity for manual or binary-level follow-up; no PE/ELF inspection is performed in v1.3.
- **CVE coverage depends on declared metadata.** `pip-audit` reads `Requires-Dist` from each archive's `METADATA`. Packages without metadata (loose source, malformed wheels) cannot be CVE-audited.
- **Windows-only paths.** Venv layout and binary extension lookups assume a Windows host. Cross-platform support is not in scope for v1.3.
- **First-run network dependency.** Scanner installation requires outbound connectivity to PyPI. Subsequent runs are offline-capable, but `pip-audit` continues to query the PyPI Advisory Database on every run unless an offline feed is staged.
- **Script directory must be writable.** v1.3 writes the venv and run logs next to the script. If the script is installed on read-only media, scans will fail at venv creation.

## Security & Operational Notes

- The scanner never executes submitted code. Tarball fallback uses a Python helper script that is written to `$env:TEMP`, invoked, and deleted in a `finally` block. The temp `.zip` file used for `.whl`/`.egg` expansion is similarly removed in a `finally` block.
- **Extracted package contents do not linger in the submission folder.** All archive extraction happens under `%TEMP%\python-scanner-<timestamp>\`, and the entire timestamped staging directory is removed at the end of every successful run (Step 8 of main execution). The submission folder receives only the summary report.
- The scanner venv is isolated from the system Python install. Removing `<script-dir>\.scan-venv\` is sufficient to reset scanner state.
- All temp JSON output files (Bandit, detect-secrets, pip-audit) are deleted in `finally` blocks regardless of scan outcome.
- `Set-StrictMode -Version Latest` and `$ErrorActionPreference = 'Stop'` are enabled at module scope to fail fast on undefined variables and non-terminating errors. EAP is temporarily relaxed to `Continue` only around native-command invocations to avoid false positives from stderr progress output, and is restored immediately afterwards (see *PowerShell 5.1 robustness* above).

## Testing & Validation

### Automated Tests

A Pester v5 test suite lives in `tests/` alongside the production script.

**How to run:**
```powershell
.\tests\Run-Tests.ps1
```

**Prerequisites:** Pester 5.x (`Install-Module Pester -MinimumVersion 5.0 -Scope CurrentUser`); Python 3 on PATH for the `Compare-Versions` integration tests (all other tests run without Python).

**Coverage (40 tests):**

| Describe block | Type | What is covered |
|---|---|---|
| `Compare-Versions` | Integration | 17 cases: exact match, semver above/below, PEP 440 pre-releases (`rc1`, `a1`, `b2`, `.dev0`), post-releases (`.post1`), local-version (`+local.1`), epochs (`2!1.0.0`), empty inputs, garbage strings, missing executable — all without Python skipped with a clear message |
| `Get-RiskLevel` | Unit | HIGH / MEDIUM / LOW / CLEAN ladder; HIGH-wins-over-MEDIUM in mixed array; empty array returns CLEAN |
| `Get-PackageUnits archive-extension classification` | Unit | `.whl`, `.egg`, `.zip`, `.tgz` → `simpleArchiveExts`; `.tar.gz` → `compoundArchiveSuffs`; entry counts verified |
| `Fixture corpus manifest` | Contract | Deterministic fixture corpus and manifest under `tests/fixtures/`, generated by `build_fixtures.py`; asserts manifest schema version `1.3`, fixture paths/layout, declared analyzer text-finding expectations, declared JSON summary expectations, and declared SBOM presence/component-count expectations |
| `Find-Python` | Unit (mocked) | All three candidates fail via `Mock Get-Command` → returns `$null`; fake `py.cmd` on PATH returns "Python 3.11.4" → returns `'py'` |

The production script is guarded at its main execution block (`if ($MyInvocation.InvocationName -ne '.')`) so dot-sourcing from the test file loads all function definitions without triggering the scan workflow. Tests write no files to `<script-dir>\.scan-venv`, `<script-dir>\logs`, or `<scan-root>\.reports`; any transient filesystem fixtures use Pester's `$TestDrive`.

The deterministic fixture corpus lives at `tests/fixtures/` and is regenerated by `tests/fixtures/build_fixtures.py`; its `manifest.json` uses schema version `1.3` and is consumed by the Pester suite to assert the manifest contract: fixture paths/layout, declared expected text findings, declared JSON summary shape expectations, and declared SBOM presence/component counts for dependency-bearing archives.

### Manual validation

Manual validation performed against:

- Clean wheels (expected: `CLEAN` verdict).
- Wheels containing intentional `eval`/`exec`/`pickle` patterns (expected: Bandit findings; severity is version-dependent and represented in the fixture manifest).
- Source files with embedded API keys (expected: detect-secrets `HIGH`).
- Wheels declaring known-vulnerable dependencies (expected: pip-audit CVE finding).
- Wheels containing `.pyd` extensions (expected: native binary `MEDIUM` finding).
- Tarballs (`.tar.gz`, `.tgz`) on hosts both with and without system `tar` available.
- Repeat scans against the same scan folder (expected: venv reuse, fresh staging, fresh report+log per run).
- Multiple submission folders processed in sequence (expected: a single shared venv, distinct timestamped logs and staging directories per run, no cross-contamination of staging contents).

`Compare-Versions` was exercised via a 16-case Python harness covering exact match, simple semver above/below, PEP 440 pre-releases (`rc1`, `a1`, `b2`, `.dev0`), post-releases (`.post1`), local-version identifiers (`+local.1`), epochs (`2!1.0.0`), empty inputs, and intentionally garbage version strings. All 16 cases produced the expected result. Seven cases now return a different (correct) result vs the prior `[System.Version]` implementation — primarily PEP 440 pre-releases that previously fell through to "assume OK" and unparseable strings that now fail closed.

The PowerShell 5.1 robustness pass was validated by static review: every native-command invocation in the script (Python `-m venv`, pip upgrade, pip install per-tool, system tar, Python tarfile fallback, bandit, detect-secrets, pip-audit, and the embedded PEP 440 comparison call) was confirmed to use the `$prevEAP / Continue / restore` pattern, every `.Count` read on a pipeline result was confirmed to be wrapped in `@(...)`, and the detect-secrets parser was confirmed to use the defensive `PSObject.Properties['results']` check before reading `$raw.results`.

See the *Automated Tests* subsection above for the Pester test suite that was added post-GA.

## Backwards Compatibility

**Upgrading from v1.2 → v1.3** requires no migration. The installation layout, output paths, and lifecycle semantics are unchanged. The only modification to the production script is the `InvocationName` guard wrapping the main execution block, which has no effect on direct invocation.

**Upgrading from pre-v1.2 drafts (`1.0`, `1.0-PEP440`) → v1.3** requires the same steps as the v1.2 migration:

1. Delete the old `.scan_workspace\` directory from any prior scan folder.
2. Move the script to its permanent home (the directory that will host the shared venv and log archive).
3. Run once with `-AutoInstall` to create the venv at `<script-dir>\.scan-venv`.

There is no automatic migration from the pre-v1.2 layout; a clean re-install is faster and safer.
